# リスク評価

## 主要リスク

### 1. external_id と common_user_id の混同

現行の `external_id` は代理店同期API用です。

これを共通ユーザーIDとして使うと、代理店ID・人物ID・外部サービスIDが混ざります。

対策:

- `common_user_id` は別管理にする
- `service_user_mappings` を追加する
- 既存APIの `external_id` は変更しない

### 2. 外部サービス起点登録の取りこぼし

ショッピングカートや戦国パスポートで先に登録されたユーザーが、代理店システムに戻ってこないと紹介関係が失われます。

対策:

- 外部サービスから代理店システムへ `common_user_id` と `service_user_id` を送るAPIを追加する
- 紹介トークンを外部サービスへ引き継ぐ
- 登録完了時に `agency_customer_relations` を作成する

### 3. 紹介元の上書きルール未定義

既に紹介元があるユーザーが別代理店のLPやカート経由で登録した場合、紹介元を変更するかどうかが問題になります。

未決定事項:

- first touch 固定
- last touch 優先
- 管理者承認で変更
- サービス別に優先順位を持つ

初期実装では、既存関係がある場合は自動上書きしない方針が安全です。

### 4. 個人情報の重複・照合

メール、電話、ウォレットアドレスで同一人物を推定する場合、誤結合のリスクがあります。

対策:

- 自動結合は `common_user_id` またはサービス側確定IDがある場合に限定する
- メール・電話は補助情報として扱う
- 手動確認フローを後続で用意する

### 5. 既存LP・問い合わせ導線への影響

LPと問い合わせは既に運用中です。

対策:

- 初期フェーズでは既存 `contact.php` の挙動を変えない
- 追加カラムがある場合のみ保存する互換実装にする
- feature flag がOFFなら新機能は動作しない

### 6. 外部APIの二重送信

外部サービスから同じ登録イベントが複数回来る可能性があります。

対策:

- `Idempotency-Key` を受け付ける
- `integration_idempotency_keys` に保存する
- 同一キーは同一レスポンスを返す

### 7. 連携障害時の調査困難

外部サービス連携は失敗時に原因が追いづらくなります。

対策:

- `integration_event_logs` を追加する
- リクエスト、レスポンス、HTTPステータス、エラーを保存する
- 管理画面で確認できるようにする

## 実装上の安全策

- 既存APIは変更しない
- 新機能は `/api/v2/*` に追加する
- DB変更は追加のみ
- feature flag は初期OFF
- ZIP更新前提でマイグレーションを分割する
- ロールバック時に既存機能へ影響しない構造にする

## 実装前に決める必要がある事項

1. `common_user_id` の発行元
2. 紹介元の上書きルール
3. 外部サービス側から送る必須項目
4. カート購入時の紹介トークン引き継ぎ方法
5. パスポート登録時の紹介トークン引き継ぎ方法
6. 売上連携を初期フェーズに含めるか
