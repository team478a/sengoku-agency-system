# PHP/API/React 段階移行方針

作成日: 2026-07-20
対象: 戦国経済圏 代理店システム

## 目的

この文書は、現在の PHP 製代理店システムを維持しながら、将来的に React などのフロントエンドへ安全に移行できる構造へ整備するための方針です。

結論として、現時点での全面 React 化は行いません。まず PHP 側を「業務ロジック/API/画面」に分離し、外部システム連携と共通顧客 HUB の中核として安定させます。

## 基本方針

1. 既存 PHP 画面は当面維持する
2. DB、認証、代理店階層、共通ID、外部連携、SSO は PHP バックエンド側に集約する
3. 新規機能は可能な限り API 経由で利用できる形にする
4. 管理画面のうち複雑な画面から React 化できるようにする
5. LP 配信は SEO、AIO、表示速度を優先し、当面 PHP テンプレート方式を維持する

## PHP に残すもの

以下は PHP 側に残すべき領域です。

- DB 接続、マイグレーション、バックアップ
- 管理者・代理店ログイン
- 権限判定
- 代理店階層管理
- 共通顧客ID管理
- 外部連携API
- SSO 署名・JWT 発行
- LP ルーティング
- LP テンプレート描画
- 問い合わせ登録
- 通知送信
- 操作ログ、ログインログ、連携ログ

理由は、これらがシステム全体の信頼性とデータ整合性に直結するためです。

## API 化を優先するもの

React 化より前に、以下を API として整理します。

- 共通顧客の作成・解決
- 外部システムアカウント紐づけ
- 代理店紹介・招待URLの確認
- 代理店階層の取得
- 代理店活動状況の取得
- LP 成果、PV、問い合わせ、LINEクリックの取得
- 外部連携先管理
- SSO 連携先管理
- ログイン記録取得
- 操作ログ取得

これにより、PHP 画面と React 画面が同じデータを使えるようになります。

## React 化に向いている画面

最初に React 化するなら、以下の管理画面が適しています。

- 共通顧客HUB
- 代理店活動分析
- 下位代理店活動ダッシュボード
- 外部API連携設定
- SSO連携設定
- ログイン記録
- 操作ログ
- 連携イベント再送画面

これらは検索、絞り込み、一覧更新、詳細表示、グラフ表示が多く、React のメリットが出やすい画面です。

## React 化を急がないもの

以下は当面 PHP のままで問題ありません。

- LP 表示
- 問い合わせフォーム
- 招待URLの公開フォーム
- パスワード再発行画面
- 基本的なログイン画面
- ZIPアップデート画面

特に LP は SEO、AIO、SNS シェア、表示速度が重要なので、React SPA 化は慎重に判断します。

## 段階移行フェーズ

### Phase 1: PHP 内部整理

- 共通処理を `includes/functions.php` に集中しすぎない
- API 用の共通レスポンス形式を整える
- 権限判定、入力検証、ログ記録を共通化する
- 既存 PHP 画面の挙動は維持する

### Phase 2: API 基盤整備

- `/api/v2` を継続利用しつつ、共通顧客HUB用 API を拡張する
- エラー形式を統一する
- idempotency key を必要な API に導入する
- API キー、SSO、外部連携ログを接続先単位で管理する

### Phase 3: 共通顧客HUBの確立

- `common_user_id` を中心に、5システムのユーザーを紐づける
- 代理店由来ユーザーと一般ユーザーを区別する
- 登録時紹介者、現在担当者、販売担当者を分離する
- 自動統合せず、曖昧な紐づけは確認対象にする

### Phase 4: React 管理画面の追加

- PHP 管理画面を残したまま React 管理画面を追加する
- 最初は読み取り中心のダッシュボードから始める
- 更新系は API と権限チェックが安定してから移行する

### Phase 5: 画面単位で置き換え

- 共通顧客HUB
- 活動分析
- 外部API連携
- SSO連携

上記の順で、画面単位に置き換えます。

## やらないこと

- 既存 PHP を一括で React に置き換えない
- LP をすぐ SPA 化しない
- DB 構造を大きく破壊しない
- 既存代理店階層、問い合わせ、LP URL を壊さない
- 共通IDを曖昧な情報だけで自動統合しない

## 次に実施する作業

1. 共通顧客HUBの現行差分レポートを作成する
2. API 一覧を「現行」「追加予定」「外部公開」に分類する
3. DB 追加案を作成する
4. React 化候補画面を優先順位づけする
5. まず PHP API と管理画面を安定させる

## 判断基準

React 化は目的ではありません。目的は、代理店システムを複数システムの中心 HUB として安全に運用できる状態にすることです。

そのため、最優先は以下です。

- データ整合性
- 外部連携の安定性
- 代理店階層の継承
- 共通顧客IDの信頼性
- 運用者が迷わない管理画面
- 障害時に復旧しやすい構造

React は、この基盤が整った後に、運用性を高めるための選択肢として導入します。
