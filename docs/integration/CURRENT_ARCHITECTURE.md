# 現行アーキテクチャ整理

## 目的

このドキュメントは、代理店システムを将来の共通ID・紹介関係・外部サービス連携の中心として拡張する前提で、現行構造を整理するものです。

重要な前提は、ユーザー登録の入口が代理店システムだけではないことです。

- 代理店システムから登録するユーザー
- ショッピングカートから購入・会員登録するユーザー
- 戦国パスポートから登録するユーザー
- 将来追加される別サービスから登録するユーザー

これらを後から同一人物・同一紹介関係として紐づける必要があります。

## 現行システムの役割

現行の代理店システムは、以下を管理しています。

- 代理店マスター
- 代理店階層
- LP URL
- プロジェクト別LP
- 問い合わせ
- 招待URL
- 外部代理店同期
- 外部サイトごとのAPIキー
- SSO連携
- ログイン記録
- 代理店活動レポート

## 現行の代理店識別子

現行の主要な識別子は以下です。

| 項目 | 用途 | 備考 |
|---|---|---|
| `agents.id` | DB内部ID | 内部参照用。外部公開キーにはしない |
| `agents.agent_code` | LP URL・代理店識別 | 既存の公開用代理店コード |
| `agents.external_id` | 外部代理店同期用ID | 既存APIで利用中 |
| `agents.parent_id` | 階層管理 | 親代理店を `agents.id` で参照 |

## common_user_id との違い

`common_user_id` は現行には存在しません。

`external_id` は外部代理店同期のためのIDであり、共通ユーザーIDとして使い回すべきではありません。

今後は以下のように分けます。

| ID | 意味 |
|---|---|
| `common_user_id` | 全サービス共通の人物ID |
| `agency_id` / `agent_code` | 代理店としての業務ID |
| `service_user_id` | 各外部サービス内のユーザーID |
| `external_id` | 既存の代理店同期API用ID |

## 外部サービス起点の登録

今後問題になるのは、外部サービス側で先に登録されたユーザーです。

例:

1. ユーザーがショッピングカートで購入・会員登録する
2. その購入が、どの代理店の紹介なのかを判定する
3. カート側のユーザーIDを `common_user_id` に紐づける
4. 代理店システム側に紹介関係を保存する
5. 戦国パスポート・ウォレットなどでも同じ人物として扱う

そのため、代理店システムには外部サービスから登録・照会できる v2 API が必要です。

## 現行機能との関係

現行機能は壊さず、追加方式で拡張します。

- 既存LP URLは維持する
- 既存問い合わせ導線は維持する
- 既存代理店同期APIは維持する
- 既存SSOは維持する
- 新しい共通ID連携は `/api/v2/*` として追加する

## 方針

代理店システムを中心ハブにする場合、最初に実装すべき責務は以下です。

- 共通IDの保存
- 外部サービスIDとのマッピング
- 紹介関係の保存
- 紹介トークンの検証
- 代理店階層の参照
- 連携ログの保存
- 冪等性の担保
- feature flag による段階公開
